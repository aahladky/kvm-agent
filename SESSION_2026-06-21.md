## 1. Real-world test — an actual Excel practice sheet
- Drove a downloaded "Data for Practice" workbook live: keyboard-launched Excel, **grounded +
  clicked "Blank workbook"** and the **total cell** (UI-TARS q4, dead-on), and filled the
  **Profit / Profit% / Weekday** formula columns down the dataset via Go To + Ctrl+Enter.
- Bug found + recovered live: the table starts at **column B** (column A is a spacer), so the
  first fill was off by one and clobbered Selling Price with `#VALUE!`. Caught it from the
  screen, **undid**, remapped B:J, and fixed a percent-format double-count. Left correct:
  Profit `=F5-E5`, Profit% `=(F5-E5)/E5`, Weekday `=TEXT(date,"dddd")`.
- Honest split: planning + perception (reading the sheet, diagnosing the spacer column) was
  **Claude**; UI-TARS did only the two grounding clicks; data entry was deterministic keyboard.

## 2. All-local planner path (tested via HF cloud as a stand-in for the B580)
- Added **`HFPlanner`** to `planner.py` — same OpenAI-compatible contract as `LocalPlanner`,
  pointed at HF's router/endpoint, token auto-resolved (`huggingface-cli login`/`HF_TOKEN`).
  Porting to the B580 later is just a `base_url` change.
- Validated end-to-end (ping → text plan → vision plan) with **Qwen3-VL-8B**, then swept
  8B/30B/72B/235B + Gemma-3-27B. Findings: easy multi-app decomposition is solved even at 8B;
  **reading exact cell refs off a dense screenshot is the hard part for all sizes** (perception,
  not planning); scale buys better *strategy*, not better *perception*.
- Three planner fixes (in `planner.py`): `max_tokens`→4000 (fixed 72B JSON truncation), a
  **column-fill idiom** (killed Gemma's 89-step unroll + unsupported drag), and reinforced
  **keyboard-first** (stopped 235B/Gemma reaching for keypad clicks). After the fixes, 235B
  produced the exact hand-written fill plan and even the 8B got the idiom + correct columns.
- (The HF test/sweep scripts were scratch and have been removed; `HFPlanner` stays in `planner.py`.)

## 3. Open WebUI front end — `agent_server.py`
- Exposes the agent as an **OpenAI-compatible streaming model** (`computer-use-agent`). Type a
  goal in Open WebUI → it runs `run_goal(executive, planner)` on the rig and **streams live
  progress** into the chat. One task at a time; `--mock` mode for safe wiring; planner via env
  (default HF Qwen3-VL-8B); executor/verifier on laptop Ollama; HID+capture on the desktop.
- Added an `on_event` progress hook to `executive.run_plan` + `planner.run_goal`.
- **Verified wired**: mock streamed into the user's Open WebUI (laptop `:8080` → desktop
  `192.168.0.184:8088`, no firewall issue). Setup in `README_openwebui.md`. Server hardened to
  **fail fast with a clear "Pico OFFLINE" message** (3s) instead of a 30s hang.
- ⚠ The real end-to-end (Open WebUI goal → live rig) was **not yet completed** — cut off when
  the Pico dropped (below). The server is running in real mode, ready once the Pico is back.

## 4. Rig firmware + robustness (⚠ NEEDS REFLASH — edited but not flash-tested from here)
- `code.py`: **F-key support (F1–F12)** — `alt+f4` etc. were silently dropped before.
- `code.py`: **WiFi self-heal watchdog** — the serving loop now detects a dropped link and

  long gaps between commands don't drop it). This is the fix for the idle "Pico went to sleep".
- `pico_env.py`: don't **orphan the camera** if the Pico boot fails.

## 5. Pico-offline incident (open)
- Mid-session the Pico went unreachable (no ping, no connect on `192.168.0.183:8000`). The rig
  is on a **Windows** host; prime suspect is **USB selective suspend / per-device power-off**
  cutting the bus-powered Pico, vs a pure WiFi drop.
- Diagnostics added (in `tools/`): `pico_diag_windows.py` (COM ports, boot_out.txt, USB-suspend
  setting, recent USB/power events), `pico_serial_log.py` (timestamped console capture),
  `wol.py` (WoL sender + `wake_and_wait()` for optional server auto-wake).
- Fixes: power-cycle the Pico to recover **now**; reflash `code.py`; on the Windows host disable
  USB selective suspend (`powercfg …`) **and** uncheck Device Manager → the Pico's USB device →
  Power Management → "Allow the computer to turn off this device to save power".
